# Диаграмма компонентов

Для реализации варианта использования «Создание и публикация портфолио» задействованы следующие компоненты:

- Контроллер портфолио принимает HTTP-запросы от веб-приложения.
- Сервис портфолио реализует бизнес-логику работы с проектами и навыками пользователя.
- Репозиторий портфолио обеспечивает доступ к данным портфолио в базе данных.
- Объектное хранилище используется для загрузки изображений и других медиафайлов проектов.

Такое разделение позволяет реализовать принципы SOLID и Separation of Concerns.

# Диаграмма последовательностей

Участники:
- Пользователь
- Веб-приложение
- Контроллер портфолио
- Сервис портфолио
- Репозиторий портфолио
- База данных

**Последовательность взаимодействия**

1. Пользователь в веб-приложении нажимает кнопку «Добавить проект».
2. Веб-приложение отправляет HTTP-запрос на сервер.
3. Контроллер портфолио принимает запрос и валидирует входные данные.
4. Контроллер вызывает сервис портфолио.
5. Сервис портфолио формирует доменную модель проекта.
6. Репозиторий портфолио сохраняет данные в базе данных.
7. База данных возвращает подтверждение сохранения.
8. Сервер отправляет успешный ответ клиенту.
9. Пользователь публикует портфолио, изменяя его статус на публичный.

Диаграмма последовательностей отражает процесс взаимодействия компонентов системы при создании и публикации портфолио. Пользователь через веб-приложение инициирует создание проекта, после чего запрос проходит через контроллер, сервис и репозиторий до уровня базы данных. Аналогичным образом осуществляется публикация портфолио путём изменения его статуса.

# Применение основных принципов разработки

## Принцип KISS (Keep It Simple, Stupid)
Код (сервер, контроллер)
```dsl
// controllers/portfolio.controller.js

class PortfolioController {
  constructor(portfolioService) {
    this.portfolioService = portfolioService;
  }

  async createProject(req, res) {
    const project = await this.portfolioService.createProject(req.body);
    res.status(201).json(project);
  }
}
```
Метод createProject выполняет одно простое действие: принимает данные и передаёт их в сервис.
Отсутствуют сложные условия, вложенные конструкции и избыточная логика, что соответствует принципу KISS.

## Принцип DRY (Don’t Repeat Yourself)
Код (сервис)
```dsl
// services/portfolio.service.js

class PortfolioService {
  constructor(portfolioRepository) {
    this.portfolioRepository = portfolioRepository;
  }

  async createProject(data) {
    return this.portfolioRepository.saveProject({
      name: data.name,
      description: data.description,
      portfolioId: data.portfolioId
    });
  }
}
```
Код (репозиторий)
```dsl
// repositories/portfolio.repository.js

class PortfolioRepository {
  constructor(db) {
    this.db = db;
  }

  saveProject(project) {
    return this.db.query(
      'INSERT INTO projects SET ?',
      project
    );
  }
}
```
Логика сохранения проекта реализована один раз в репозитории и переиспользуется сервисом.
Контроллеры и сервисы не содержат дублирующихся SQL-запросов, что соответствует принципу DRY.

## Принцип YAGNI (You Aren’t Gonna Need It)
Код (репозиторий — только нужные методы)
```dsl
class PortfolioRepository {
  constructor(db) {
    this.db = db;
  }

  saveProject(project) {
    return this.db.query('INSERT INTO projects SET ?', project);
  }

  updateStatus(id, isPublic) {
    return this.db.query(
      'UPDATE portfolios SET isPublic = ? WHERE id = ?',
      [isPublic, id]
    );
  }
}
```
В репозитории отсутствуют методы удаления, архивирования и версионирования портфолио, так как они не требуются для текущего варианта использования.
Реализован только минимально необходимый функционал, что соответствует принципу YAGNI.

## Принципы SOLID
### SRP — Single Responsibility Principle
Код
```dsl
// controllers/portfolio.controller.js
class PortfolioController {
  async publishPortfolio(req, res) {
    await this.portfolioService.publish(req.params.id);
    res.sendStatus(200);
  }
}
```
```dsl
// services/portfolio.service.js
class PortfolioService {
  async publish(id) {
    return this.portfolioRepository.updateStatus(id, true);
  }
}
```
Контроллер отвечает только за HTTP-взаимодействие, а сервис — за бизнес-логику публикации портфолио.

### OCP — Open/Closed Principle
Код
```dsl
class PortfolioService {
  async createProject(data) {
    return this.portfolioRepository.saveProject(data);
  }
}
```
Для добавления нового типа проекта достаточно расширить модель данных, не изменяя существующие методы сервиса.

### LSP — Liskov Substitution Principle
Код
```dsl
class PortfolioService {
  constructor(portfolioRepository) {
    this.portfolioRepository = portfolioRepository;
  }
}
```
Сервис не зависит от конкретной реализации репозитория.
При необходимости репозиторий можно заменить (например, MySQL → PostgreSQL) без изменения логики сервиса.

### ISP — Interface Segregation Principle
Код
```dsl
class PortfolioRepository {
  saveProject(project) {}
  updateStatus(id, isPublic) {}
}
```
Репозиторий содержит только методы, которые реально используются.
Отсутствуют «толстые» интерфейсы с неиспользуемыми операциями.

### DIP — Dependency Inversion Principle
Код
```dsl
const controller = new PortfolioController(
  new PortfolioService(
    new PortfolioRepository(db)
  )
);
```
Контроллер и сервис получают зависимости извне, а не создают их самостоятельно.
Это снижает связанность компонентов и упрощает тестирование.

# Дополнительные принципы разработки

### BDUF (Big Design Up Front)

Принцип BDUF предполагает детальное и полное проектирование архитектуры системы до начала её реализации.

В рамках проекта PortfolioPro данный принцип не был применён в полной мере. Несмотря на то, что на раннем этапе была сформирована общая архитектурная модель (контейнеры и компоненты в нотации C4), детальная проработка всех модулей и сценариев заранее не выполнялась.

Причины отказа от BDUF:
- требования к функциональности сервиса могут изменяться в зависимости от обратной связи пользователей;
- проект ориентирован на постепенное расширение возможностей (аналитика, рекомендации, интеграции);
- чрезмерно детальное проектирование на раннем этапе могло бы замедлить разработку.

Вместо BDUF был выбран итеративный подход, при котором архитектура уточняется и развивается по мере появления новых требований.

### SoC (Separation of Concerns)

Принцип разделения ответственности был применён в рамках всего проекта и является одним из ключевых архитектурных принципов.

Разделение ответственности реализовано на нескольких уровнях:
- на уровне архитектуры — разделение на фронтенд, бэкенд, базу данных и внешние сервисы;
- на уровне серверной части — выделение контроллеров, сервисов и репозиториев;
- на уровне клиентской части — разделение интерфейсных компонентов по функциональному назначению (профиль, портфолио, резюме, аналитика).

Применение принципа SoC позволяет:
- упростить сопровождение и тестирование системы;
- изолировать изменения в отдельных модулях;
- повысить масштабируемость проекта.

### MVP (Minimum Viable Product)

Принцип MVP был применён при формировании первоначального функционала проекта.

На первом этапе разработки реализован минимально жизнеспособный продукт, включающий:
- регистрацию и профиль пользователя;
- добавление проектов и навыков;
- базовую визуализацию портфолио;
- генерацию PDF-резюме;
- публикацию портфолио по публичной ссылке.

Функции расширенной аналитики, рекомендаций навыков и интеграции с внешними платформами рассматриваются как дальнейшие этапы развития.

Использование принципа MVP позволяет:
- сосредоточиться на ключевой ценности сервиса;
- сократить время выхода первой версии;
- обеспечить возможность получения пользовательской обратной связи.

### PoC (Proof of Concept)

Принцип PoC был применён точечно в рамках проекта.

В частности, в качестве доказательства концепции рассматривается:
- анализ навыков пользователя на основе данных рынка труда;
- интеграция с внешними источниками вакансий (например, HH.ru API).

На данном этапе данные механизмы реализуются в виде прототипов, позволяющих проверить:
- применимость выбранных источников данных;
- корректность сопоставления навыков и требований вакансий;
- потенциальную ценность аналитики для пользователей.

Полноценная реализация данных модулей планируется только после подтверждения жизнеспособности концепции.
